# 中文小说主编台设计文档

- 日期：2026-03-16
- 状态：Draft
- 目标读者：内部 2 到 3 人小说创作团队
- 设计原则：Anthropic 风格的 `workflow-first`、`state-first`、`human-in-the-loop`

## 1. 背景与目标

本产品是一款内部使用的中文小说创作系统，核心目标不是做一个“会聊天的写作 AI”，而是做一个能够长期支撑长篇小说生产的主编台。

团队当前最优先解决的问题按顺序为：

1. 长篇失控
2. 写章慢
3. 起书慢
4. 改稿慢

第一版优先服务网文写作场景，但底层设计必须保留后续扩展到更复杂创作场景的余地。

## 2. 设计原则

本系统明确对齐 Anthropic 公布的 agent 设计方法，优先采用简单、透明、可控的模式。

原则如下：

1. 能用工作流解决，就不用多 agent。
2. 能用结构化状态解决，就不用让模型隐式记忆。
3. 能用单次调用完成，就不用长链路自治。
4. 重大状态变更必须可见、可审批、可追踪。
5. 只有在简单方案无法满足需求时，才增加 agent 复杂度。

## 3. 产品定义

第一版产品定义为：

一款面向 2 到 3 人内部团队的中文小说主编台。前台提供可见的项目状态、章节工作台和控盘能力；后台以工作流引擎为核心，辅以少量必要智能能力，帮助团队从立项、大纲、章节生产到长篇纠偏进行持续创作。

这不是一个“万能聊天框”，而是一个围绕小说项目状态运作的生产系统。

## 4. 用户与核心场景

### 4.1 目标用户

- 内部网文创作团队
- 小规模协作，通常 2 到 3 人
- 强调连载效率与长篇稳定性

### 4.2 核心场景

1. 从题材到可开写项目的快速立项
2. 基于章节卡稳定生成章节初稿
3. 对中后期长篇进行结构化体检与纠偏
4. 对已有章节做定向重写、润色和一致性修复

## 5. 系统总体方案

### 5.1 总体架构

系统分为 6 层：

1. 主编台前端
2. 应用后端
3. 工作流与编排层
4. 模型调用层
5. 小说状态层
6. 检索与校验层

### 5.2 为什么不采用多常驻 agent 架构

第一版不采用“多个人格化 agent 自由协作”的架构。原因如下：

- 对小团队来说，复杂度收益比过低
- 长篇创作更依赖稳定状态而不是 agent 间对话
- 自由协作容易造成上下文污染和状态冲突
- Anthropic 的方法更强调简单工作流与明确边界

### 5.3 第一版智能能力形态

第一版默认只暴露一个前台写作入口，内部不强调多个并列 agent。

系统只保留三种能力模式：

1. 写作模式
2. 审校模式
3. 控盘模式

这三种模式由同一套工作流系统调度，可共享同一个模型接口实现，也允许后续按收益拆分成独立能力。第一版不要求它们在运行时表现为三个独立 agent。

## 6. 核心能力设计

### 6.1 写作模式

职责：

- 生成立项草案
- 辅助设定整理
- 生成总纲、卷纲与章节卡
- 生成章节初稿
- 执行局部改写与润色
- 回答与当前项目状态相关的自然语言问题

边界：

- 不直接静默修改硬设定
- 不直接覆盖已批准的大纲主线
- 不直接写入正式状态，默认进入待审核区

### 6.2 审校模式

职责：

- 检查角色一致性
- 检查设定冲突
- 检查章节是否偏离章节卡
- 检查伏笔是否遗漏或过期
- 检查节奏、信息增量与钩子强度

输出：

- 结构化问题列表
- 修订建议
- 可选修订稿

### 6.3 控盘模式

职责：

- 对全书或当前卷进行体检
- 识别主线偏移、支线膨胀、角色失衡、伏笔回收风险
- 给出保守修正、中度修正、强制收束三类建议

触发方式：

- 卷末检查
- 固定章节数后触发
- 用户主动发起

## 7. 小说状态库设计

状态库是系统事实源，不以聊天记录为准。

### 7.1 核心对象

每本书至少包含以下对象：

1. `project`
2. `world_rules`
3. `characters`
4. `plot`
5. `chapter_cards`
6. `drafts`
7. `foreshadow_registry`
8. `issues_and_decisions`

### 7.2 对象说明

`project`

- 书名
- 题材
- 卖点
- 文风标签
- 当前阶段
- 版本信息

`world_rules`

- 世界观规则
- 力量体系
- 势力与地点
- 硬设定
- 软偏好
- 禁忌设定

`characters`

- 角色基本信息
- 目标与动机
- 成长弧线
- 关系网
- 说话风格
- 当前状态

`plot`

- 总纲
- 卷纲
- 主线与支线
- 高潮点
- 回收计划

`chapter_cards`

- 章节编号
- 本章目标
- 冲突
- 人物推进
- 信息增量
- 伏笔埋设或回收
- 结尾钩子
- 禁止偏离项

`drafts`

- 初稿
- 修订稿
- 定稿
- 版本历史

`foreshadow_registry`

- 伏笔条目
- 首次出现章节
- 计划回收章节
- 当前状态
- 关联角色或事件

`issues_and_decisions`

- 审校问题
- 重大改动
- 审批记录
- 生效范围
- 决策原因

### 7.3 状态写入规则

必须严格限制写权限：

- 写作模式只允许提交草案对象，不直接提交正式状态
- 审校模式只允许提交问题、标注与修订建议
- 控盘模式只允许提交结构风险报告与修正方案
- 工作流执行器只允许按既定规则写入，不允许自行决定业务含义
- 涉及硬设定、角色核心动机、主线方向的修改必须经人工审批

状态流转固定为：

`proposal -> review -> approve -> commit`

### 7.4 审批模型

审批是第一版的核心控制面，不允许模糊处理。

审批对象分为两类：

1. `对象级审批`
   用于单个状态对象的变更，例如角色卡、世界观规则、章节卡、正文定稿。
2. `流程级审批`
   用于一次工作流产出的整组变更，例如“卷纲调整 + 后续 8 章章节卡重排”。

审批角色：

- `主编`
  有权审批所有重大变更。
- `作者`
  可发起草案、接受非重大正文修订、提交待审批变更。

审批规则：

- 硬设定、角色核心动机、主线方向、重大伏笔调整，必须由主编审批
- 普通章节定稿可由作者确认，但如果系统标记为高风险章节，则升级为主编审批
- 流程级审批通过后，系统按对象逐项提交 `commit`

审批结果必须记录：

- 审批人
- 审批时间
- 审批对象
- 原始 proposal
- 最终 commit 版本
- 审批理由或备注

## 8. 工作流设计

### 8.1 工作流 A：立项到开写

流程：

1. 用户输入题材、关键词、参考风格与禁忌
2. 生成立项卡
3. 生成世界观与角色草案
4. 生成总纲与卷纲
5. 生成前 10 到 20 章章节卡
6. 审校检查结构一致性
7. 主编对立项卡、设定草案、总纲、前置章节卡做流程级审批
8. 审批通过后按对象逐项落库

### 8.2 工作流 B：章节生产

流程：

1. 选择目标章节
2. 系统组装最小上下文包
3. 生成章节初稿
4. 审校检查
5. 可选改写
6. 作者或主编确认章节定稿
7. 系统按规则更新正文、角色状态、伏笔状态与决策日志

### 8.3 工作流 C：长篇纠偏

流程：

1. 发起全书或当前卷体检
2. 输出结构风险报告
3. 生成 3 类纠偏方案
4. 用户选择方案
5. 主编对修正方案做流程级审批
6. 更新后续大纲或章节卡
7. 写入决策日志

### 8.4 自动化边界

可以自动完成：

- 立项草案
- 大纲草案
- 章节卡
- 初稿
- 审校报告
- 修订建议

不能自动最终确认：

- 硬设定变更
- 核心角色动机变更
- 主线重定向
- 重要伏笔删除或延后
- 已定稿事实覆盖

### 8.5 最小上下文包规则

章节生产时，系统必须按固定顺序组装上下文，不允许自由拼接。

默认上下文包包含：

1. 当前章节卡
2. 当前卷纲摘要
3. 本章相关角色卡
4. 与本章直接相关的世界观硬设定
5. 最近 3 到 5 章摘要
6. 与本章相关的伏笔条目
7. 文风规则与禁止项

冲突优先级固定为：

1. 硬设定
2. 已批准的角色当前状态
3. 已批准的卷纲与章节卡
4. 最近章节摘要
5. 当前草稿局部内容

生成前校验：

- 如果缺少章节卡，不允许直接生成正式初稿
- 如果硬设定与章节卡冲突，先输出冲突报告
- 如果角色状态缺失，标记为上下文不足并要求补档

历史内容注入规则：

- 默认只注入最近 3 到 5 章摘要，不直接注入完整正文
- 只有用户显式要求或系统判定为强连续章节时，才附加上一章正文全文
- 检索结果必须带来源对象标识，方便回溯

## 9. 产品界面设计

第一版只做 3 个核心页面，避免过度铺陈。

### 9.1 项目页

展示：

- 当前书项目信息
- 当前卷进度
- 最近完成章节
- 风险提示
- 待回收伏笔数量
- 推荐下一步动作

### 9.2 章节工作台

结构建议为三栏：

- 左栏：章节卡、相关设定、相关角色
- 中栏：正文编辑与生成区
- 右栏：审校结果、改写命令、风险提示

高频动作：

- 生成初稿
- 局部重写
- 收紧节奏
- 增强钩子
- 对齐章节卡
- 确认落库

### 9.3 控盘页

展示：

- 总纲与当前进度偏差
- 角色活跃度
- 支线膨胀情况
- 伏笔回收状态
- 卷级节奏风险
- 发起体检入口

## 10. 技术方案

### 10.1 技术原则

- 先工作流，后自治
- 先显式状态，后隐式记忆
- 先单模型接口封装，后多模型优化
- 先明确审批点，后追求自动化

### 10.2 分层建议

`前端`

- Web 主编台

`后端`

- 项目与任务管理
- 审计日志
- 版本记录

`编排层`

- 负责任务路由与工作流执行
- 只负责编排，不直接拥有内容判断权

`模型层`

- 统一封装创作、分析、改写能力

`状态层`

- 保存结构化对象与正文版本

`检索校验层`

- 上下文组装
- 冲突检测
- 风险分析
- 审批前校验

### 10.3 pi coding agent 的定位

如果使用 pi coding agent 或同类能力，建议只放在编排层，作为：

- 工作流 orchestrator
- 任务调度器
- 确定性状态写入器的调用方

这里必须明确：

- `pi coding agent` 不是业务决策者
- 它不决定设定是否该改、剧情是否该转向
- 它只能根据已经批准的 proposal 调用确定性写入工具完成 `commit`

不建议把它直接作为小说生成主脑。

## 11. 扩展性设计

### 11.1 必须稳定的基础对象

以下对象的语义应尽量长期稳定：

- 项目
- 世界观规则
- 角色卡
- 剧情骨架
- 章节卡
- 正文版本
- 伏笔注册表
- 决策日志

### 11.2 可扩展方向

后续新增能力应以“读取既有对象并追加新工作流”为主，例如：

- 题材模板库
- 爆点增强器
- 文风迁移器
- 多书共宇宙管理
- 漫画或剧本改编
- 多模型路由策略

### 11.3 可扩展机制

建议采用：

- 注册式能力声明
- 版本化状态模型
- 可替换工作流节点
- 可替换模型路由
- 可选地把“审校模式”或“控盘模式”独立拆成专用运行时组件，但前提是收益明确

## 12. 第一版范围

### 12.1 包含

1. 项目创建与立项
2. 设定、大纲、章节卡管理
3. 章节初稿生成与局部改写
4. 审校与一致性检查
5. 长篇控盘体检
6. 状态库与审批式落库

### 12.2 不包含

1. 面向外部用户的商业化能力
2. 多租户复杂权限系统
3. 大规模协同编辑
4. 高保真视觉品牌包装
5. 完全自治的多 agent 群

## 13. 成功标准

第一版成功标准：

1. 能把一本小说从立项推进到持续章节生产
2. 能让章节生产建立在结构化章节卡之上
3. 能在中后期识别长篇失控风险
4. 能让团队成员基于统一状态库协作
5. 能为后续迭代保留清晰扩展边界

## 14. 风险与注意事项

1. 如果状态模型设计不严谨，后续功能越多越容易混乱
2. 如果把太多逻辑藏进 prompt，会破坏系统透明性
3. 如果过早引入过多 agent，会增加调试与维护负担
4. 如果缺少审批与日志，长篇纠偏会不可追溯

## 15. 结论

第一版应采用一种克制的 agent 产品设计：

- 一个主编台
- 一套清晰工作流
- 一个前台写作入口
- 同一工作流系统下的写作、审校、控盘三种能力模式
- 一套结构化小说状态库
- 审批式状态更新机制

这套方案比“多 agent 自由协作”更符合 Anthropic 的设计哲学，也更适合内部小说团队长期使用与后续升级。
