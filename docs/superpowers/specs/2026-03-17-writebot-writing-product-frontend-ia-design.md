# Writebot 创作型前端信息架构调整设计

## 1. 背景

当前仓库里的原型方向已经明确了 `workflow-first`、`human-confirmed`、`artifact-driven` 这些 agent 产品原则，但前台信息架构仍然更接近“系统控制台”，而不是“创作产品”。

这会带来两个直接问题:

- 用户首先感知到的是系统模块，而不是写作任务
- `确认`、`修复`、`控制` 被做成独立页面后，写作流被频繁切断

本设计的目标不是推翻底层 agent 能力，而是重新包装前台壳层，让产品更像成熟的创作工具，同时保留原有的状态、确认、修复与审计能力。

## 2. 设计目标

本次调整只聚焦前端产品壳层与信息架构，目标如下:

- 让用户首先感知“项目”和“章节”，而不是“调度”和“控制台”
- 把写作编辑器重新放回产品中心
- 把 agent 状态下沉为上下文能力，而不是主导航页面
- 全面汉化用户前台界面，优先使用自然、可理解的创作产品语言
- 在不增加学习成本的前提下，保留明确的人类确认入口

## 3. 核心结论

Writebot 前台应从原先的四页面 agent 原型:

- `Task Dispatch Cockpit`
- `Confirmation Center`
- `Creation Workbench`
- `Control Console`

收敛为更适合创作产品的三层结构:

1. `项目`
2. `项目概览`
3. `章节工作区`

其中:

- `确认中心` 不再作为主页面存在，而是降级为章节工作区内的内联确认层
- `控制台` 不再作为主页面存在，而是降级为修复提示卡和详情抽屉
- `Creation Workbench` 继续保留，但升级为真正的写作主界面
- `Task Dispatch Cockpit` 的职责并入项目概览页

## 4. 一级信息架构

调整后的前台推荐保留以下导航层级:

- `项目`
- `项目概览`
- `章节工作区`
- `故事设定`
- `大纲`
- `历史记录`

这套结构中，真正的主流程只有两步:

`项目概览 -> 章节工作区`

其余入口都是项目级辅助资源，而不是和写作主流程平级竞争的系统模块。

## 5. 页面一: 项目概览

### 5.1 页面定位

项目概览页是进入单个项目后的首页，但它不应被设计成后台 dashboard，而应是“创作导航页”。

用户进入这里后，应该立刻知道三件事:

- 当前项目进展到哪里
- 推荐优先推进哪一章
- 是否有少量需要先处理的事项

### 5.2 页面结构

项目概览页建议由以下区域组成:

- 顶部项目头部
  - 项目名
  - 一句当前进度摘要
  - 主按钮: `继续写作`
- 主区域
  - 章节卡片列表
- 次级信息区
  - 少量项目级提示

### 5.3 章节卡片

章节卡片是本页的核心对象。每张卡片只保留最少但足够决策的信息:

- 章节名
- 一句话摘要
- 当前状态
- 最近动作
- 入口按钮: `进入章节`

状态文案统一收敛为:

- `可继续写作`
- `待确认`
- `进行中`
- `需修复`

### 5.4 页面原则

本页不应暴露过多内部术语或系统细节，例如:

- `dispatch`
- `execution lane`
- `risk queue`
- `artifact state`

它应该更接近“作品目录 + 当前进度 + 推荐下一步”，而不是“代理系统总控页”。

## 6. 页面二: 章节工作区

### 6.1 页面定位

章节工作区是新的主战场。产品的大部分停留时间都应发生在这里。进入这一页后，用户的第一感受必须是“我正在写这一章”，而不是“我正在操作一个复杂 agent 系统”。

### 6.2 三栏结构

章节工作区建议固定为三栏布局:

- 左栏: `章节导航`
- 中栏: `正文编辑区`
- 右栏: `智能辅助`

### 6.3 左栏: 章节导航

左栏只承担导航职责:

- 显示当前项目下的章节列表
- 高亮当前章节
- 允许快速切换章节

不建议在左栏堆积复杂状态统计，否则会破坏它作为轻导航的作用。

### 6.4 中栏: 正文编辑区

中栏是绝对主角，应拥有最大的视觉权重。推荐包含:

- 章节标题
- 当前写作目标
- 轻量状态横幅
- 正文编辑器

其中状态横幅应极其克制，只在确有必要时出现，例如:

- `有 1 项内容需要你确认`
- `检测到一项连续性问题`

### 6.5 右栏: 智能辅助

右栏不是功能大全，而是上下文辅助区。默认只显示 2 到 3 张最相关的卡片，按上下文动态切换。

推荐优先级如下:

1. `当前建议`
2. `本章上下文`
3. `可用操作`

右栏的动态规则建议为:

- 刚进入章节时: 优先显示目标、约束与背景
- 正在写正文时: 优先显示续写、改写、扩写、精简
- 出现待确认事项时: 第一张卡片变为确认卡
- 出现偏离或连续性风险时: 第一张卡片变为修复建议卡

## 7. 内联确认与修复层

### 7.1 待确认

需要人类确认的动作不再跳转到单独页面，而是在当前章节上下文里处理。

推荐表现形式:

- 编辑器上方出现轻量横幅: `有 1 项内容需要你确认`
- 右栏顶部出现确认卡

确认卡只回答三件事:

- 为什么需要确认
- 会影响什么
- 下一步怎么处理

主动作只保留两个:

- `确认继续`
- `退回重做`

### 7.2 需修复

当系统检测到连续性问题、策略冲突或明显偏离时，也不应跳去独立控制台。

推荐表现形式:

- 编辑器上方出现弱提示
- 右栏顶部出现修复建议卡

当信息量超出右栏承载能力时，再从当前页面展开详情抽屉，用于承载:

- 影响范围
- 证据
- 修复建议

### 7.3 原则

确认与修复层必须同时满足三条规则:

- 不打断主写作流
- 不让用户错过高风险状态
- 不把 agent 能力做成后台控制台

## 8. 全面汉化与命名策略

前台界面统一使用中文命名，优先采用写作产品语言，而不是内部系统语言。

建议命名如下:

- `Projects` -> `项目`
- `Project Overview` -> `项目概览`
- `Chapter Workspace` -> `章节工作区`
- `Story Bible` -> `故事设定`
- `Outline` -> `大纲`
- `History` -> `历史记录`

状态命名:

- `Ready to write` -> `可继续写作`
- `Needs confirmation` -> `待确认`
- `In progress` -> `进行中`
- `Needs repair` -> `需修复`

动作命名:

- `Continue Writing` -> `继续写作`
- `Open Chapter` -> `进入章节`
- `View confirmation` -> `查看确认`
- `Approve and continue` -> `确认继续`
- `Send back` -> `退回重做`

不建议继续把以下词汇作为前台一级语言:

- `Task Dispatch Cockpit`
- `Confirmation Center`
- `Control Console`
- `execution lane`
- `artifact state`

这些表达可以保留在内部模型、代码命名和实现层，但不应继续作为面向用户的主界面语言。

## 9. 对现有原型的迁移映射

现有原型页面的职责建议重新映射如下:

- `Task Dispatch Cockpit` -> 并入 `项目概览`
- `Creation Workbench` -> 升级为 `章节工作区`
- `Confirmation Center` -> 降级为内联确认卡与抽屉
- `Control Console` -> 降级为修复建议卡与详情抽屉

迁移后的感知变化应为:

- 从“模块切换”转向“围绕章节持续工作”
- 从“系统状态中心”转向“创作中心”
- 从“单独确认页面”转向“必要时露出的上下文确认”

## 10. 非目标

本次设计不追求:

- 在前台完整暴露底层 agent 拓扑
- 把所有状态都变成常驻可见模块
- 新增复杂的项目管理层级
- 把前台做成纯聊天式写作工具

本次设计也不要求一次性定义所有视觉细节。当前阶段只锁定:

- 页面层级
- 主交互流
- 命名体系
- 现有原型的收敛方向

## 11. 实施后果

如果按本设计推进前端原型，当前实现计划需要同步调整:

- 主导航应围绕 `项目概览` 与 `章节工作区` 重组
- 原有四个独立页面的职责应重新拆分与吸收
- 确认与修复交互要改为工作区内联层
- 所有面向用户的文案要做全面汉化

这意味着后续实现应优先考虑:

- 先完成新的页面壳层与中文命名
- 再将现有原型数据重新映射到新页面
- 最后补齐右栏上下文卡片、横幅和抽屉的交互细节

## 12. 结论

Writebot 的下一版前端原型，不应继续强化“agent 控制台”形态，而应转向“创作产品”形态。

推荐落地方案是:

- 用 `项目概览` 承接项目级导航
- 用 `章节工作区` 承接核心写作体验
- 用 `内联确认与修复层` 承接 agent 能力暴露
- 用全面汉化的创作语言替换当前偏系统化的界面表达

这套方案能在不削弱底层 agent 能力的前提下，显著降低前台复杂感，并让产品体验更接近成熟的长篇写作工具。
